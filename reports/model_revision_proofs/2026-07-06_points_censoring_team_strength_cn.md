# 2026-07-06 积分截断车队强度修订证明

## 结论

这次 British GP 预测变化不能描述为“新增信息驱动”。更准确的分类是：

```text
用户指出异常
-> 触发模型链路审计
-> 发现已存在结构化数据的使用方式有缺陷
-> 修订通用特征映射
-> 使用同一批 FastF1 / 官方结构化来源重新生成诊断预测
```

用户的例子不进入模型输入，不作为车队或车手标签，也不允许变成任何实体级手调规则。

## 本次没有做的事

- 没有因为用户说某个车队强或弱，就直接调高或调低该车队数值。
- 没有写 `if team == ...` 或 `if driver == ...` 这类实体特判来迎合预期排名。
- 没有把用户的判断当作训练标签、来源证据或赛前事实。
- 没有证明模型已经具备稳定盈利 edge。

## 本次实际做的事

修订的是一个通用数据映射问题：F1 分站积分只奖励前十名，因此积分是被规则截断后的观测。如果只用每站积分重估车队强度，会把经常第 11-14 名完赛的车队和经常第 18-20 名完赛的车队都压成接近 0 分。

因此修订后的链路会同时读取：

- FastF1 截止日前的分站正赛积分；
- FastF1 截止日前的全场完赛名次；
- 最近若干站窗口；
- 全场平均水平。

然后用同一个公式对所有车队计算车队正赛速度修正。这个公式不关心车队名称，只关心来源数据里的积分和完赛名次。

## 为什么这是模型修订，不是新来源驱动

本次 diff 显示：

```text
input_changed = true
evidence_changed = false
information_intake_changed = false
probability_changed = true
```

也就是说，预测变化不是因为新抓取了网页新闻或新增了 Codex 证据，而是因为同一批结构化数据通过新的通用映射方式进入了模型。

所以它只能被标记为“模型/映射修订诊断 run”。它可以用于后续审计和回放验证，但不能被包装成“来源已经证明排名更合理”。

## 如何证明没有按用户一句话手调

当前新增的合约测试是：

```text
scripts/source_driven_contract_test.py
```

它检查两类风险：

- 核心预测更新文件里不能出现用户举例中的车队或车手 id，用来防止实体级硬编码；
- 如果预测概率变了，但没有新证据或新原始来源身份，只是 BeliefState / 映射结果变了，则默认不能注册成 latest，除非显式提供模型修订证明。

这条规则的含义是：用户反馈只能触发审计，不能直接变成预测数值。

## 当前预测效果的正确表述

允许说：

```text
这次修订让模型开始利用全场完赛名次来缓解积分前十截断问题，诊断预测发生了小幅变化。
```

不允许说：

```text
因为用户说某队应该更强/更弱，所以模型把它改到了更合理的位置。
```

也不应直接说：

```text
排名更合理，所以预测能力显著提升。
```

真正能证明预测能力提升的材料必须来自历史回放、同口径概率校准、市场基线比较和赛前 cutoff 严格一致的评估。
