# F1Predict 项目期望与关键提醒

生成日期：2026-07-05

这份文档记录用户对 F1Predict 项目的真实期望。以后每次继续实现、重构、评估或展示前，都应该先读这份文档，避免把项目做成“看起来工程很多，但预测没有变好”的系统。

## 1. 项目真正目标

目标不是做一个漂亮的 F1 展示页，也不是做一个能解释自己的玩具模型。

目标是建立一个能在事件预测中稳定寻找 edge 的系统。F1 只是第一个模拟场景：先做分站排名、分站积分、全年积分预测，再用知识截止到上周的方式回放真实比赛，检验模型有没有预测能力。

预测系统必须能回答：

- 预测了每个车手的预计完赛排名、积分、领奖台概率、退赛风险和不确定性；
- 为什么这样预测；
- 哪些信息源改变了预测；
- 哪些因素贡献最大；
- 和上一版预测相比变了什么；
- 历史回放是否更准；
- 是否真的存在可交易或可决策的 edge。

## 2. Codex/LLM 的正确角色

Codex 不是一个可以随便拿掉做 ablation 的小模块。

Codex/LLM 在这个项目里的核心作用是：把网上所有可能有用的非结构化信息检索、阅读、筛选、归一化，并转成本地结构化证据。

没有这层信息摄取，模型只能依赖公开结构化数据、历史结果和手写 seed prior。那不是完整预测，只是弱 baseline。

正确的对照不是“有 Codex vs 无 Codex”，而是：

- 结构化公开数据 baseline；
- 加入 Codex 检索和归一化后的信息摄取版本；
- 不同信息质量、不同因子类别、不同来源类别的对照。

要证明的是：Codex 摄取的非结构化信息是否让预测更好，而不是证明“没有信息也能预测”。

## 3. 用户指出的“直观结论”不是手写强约束

例如：

- 赛季初梅奔很强；
- 红牛赛季初很弱；
- 红牛最近两站有所长进；
- 法拉利明显是梅奔下面的第二梯队。

这些不是要求模型手动写死，而是要求系统从简单公开信息中自然推断出来。

如果系统连这些结论都不能识别，说明信息检索、信息理解、结构化和因子映射层有问题。

这些判断应该来自：

- 练习赛、排位赛、正赛 pace；
- 长距离速度、轮胎衰退、stint 表现；
- sector time、speed trap、直道速度、弯道表现；
- 车队升级、技术分析、车手采访、车队采访；
- 可靠性、调校、轮胎窗口、天气和赛道适配；
- 多个独立信息源的一致性；
- 市场赔率或交易价格的隐含信息。

## 4. 重要预测因素必须进入统一 ontology

所有新闻、采访、数据、技术分析，最终都应该归结到会影响比赛结果的因素上。

### 赛道因素

- 正确赛道图和赛道几何；
- 每个弯角类型：低速弯、中速弯、高速弯、直角弯；
- 短直道、长直道、DRS 或 2026 年替代技术适用区域；
- 单个弯角超车可能性、整条赛道超车难度；
- 沥青情况、轮胎磨耗、赛道演化；
- 历史气温、天气预报、赛道温度；
- 比赛时间、海拔、街道赛或传统赛道。

### 比赛随机事件

- 黄旗、安全车、虚拟安全车；
- 黄旗下免费进站窗口；
- 红旗概率；
- 退赛概率；
- 维修区损失；
- 轮胎策略选择；
- 起步、缠斗、事故、处罚。

### 赛车因素

- 动力单元实力；
- 涡轮、能量回收、部署能力；
- 底盘水平、机械抓地力、气动水平；
- 推头或甩尾倾向；
- 高低温、高低海拔表现；
- 升级包效果和验证程度；
- 低速弯、中速弯、高速弯、长直道分别的适配。

### 车手因素

- 排位单圈极限；
- 长距离能力；
- 保胎能力；
- 缠斗能力；
- 雨战能力；
- 起步能力；
- 调校能力；
- 与赛车特性的适配；
- 年龄、成长空间、心态；
- 队内地位、是否一号车手；
- 与队友关系；
- 是否能影响车队升级方向。

### 车队因素

- 历史决策水平；
- 策略偏好；
- 换胎实力；
- 领队个人能力；
- 工程团队人员流动；
- 研发方向是否匹配当前赛车问题；
- 主场、工厂距离、后勤优势。

### 杂项

- 历史赛道适配；
- 主场优势；
- 赛程位置和连续背靠背影响；
- FIA 判罚倾向如果有证据可以记录，但不能无证据强行使用。

## 5. 预测不能只看第一名

系统必须预测整场比赛，而不是只预测冠军。

前端和后端都应该关注：

- 20 名车手的排名分布；
- expected finish；
- expected points；
- win/podium/top5/top10 概率；
- DNF 风险；
- 排位和正赛可能的差异；
- 关键车手之间的 H2H；
- 车队总积分、双车进 Q3、双车积分区、双领奖台等事件；
- 全年积分预测和不确定性。

如果前端展示中车手之间概率差异极小，必须解释原因；如果解释不出来，大概率是模型错误、输入错误或展示错误。

## 6. 前端应该展示什么

前端不应该堆大量内部审计内容。

前端重点应该展示所有会影响预测结果的信息：

- 当前预测结果和不确定性；
- 每个车手的预计排名/积分/概率分布；
- 关键因子贡献；
- 信息源证据；
- 与上一版预测的变化；
- 为什么变化；
- 历史回放表现；
- 哪些输入还缺失；
- 当前预测是否可以用于 edge 判断。

前端应该是中文版本，面向人审阅和决策，而不是面向调试日志。

前端加载应该快。预测结果不应该靠页面实时重算，应该读取后端已经生成和缓存的 prediction artifact。

## 7. 不要做单点修补

如果用户指出 British Grand Prix 赛道图有问题，不代表只修 British Grand Prix。

这说明同类型数据源或映射逻辑存在系统性问题，必须统一修整整套赛道图/赛道数据源。

所有列表式、同类型数据的问题都应该优先检查整体数据源，而不是只修用户点名的一条。

## 8. 每次更新都必须影响可审计的预测链路

搭好框架之后，每一次代码更新、信息更新、权重更新都应该能回答：

- 输入数据改变了什么；
- 哪些 claim/factor 改变了；
- 哪些车手概率和预计排名改变了；
- 变化幅度是多少；
- 为什么变化；
- 历史 replay 指标是否变好；
- 如果没有影响预测，为什么没有影响。

不能再出现“做了很多事情，但前端预测结果很久不变，也不知道是模型没变、前端没更新，还是 artifact 没接上”的状态。

## 9. 评估和对照必须诚实

不能把 probe、诊断、半成品说成正式实验。

对照实验必须匹配：

- 同一知识截止时间；
- 同一赛程和结果可见性；
- 同一市场快照时间；
- 同一随机种子或足够稳定的模拟设置；
- 同一评估指标；
- 同一预测输出口径。

真正应该做的是：

- 结构化公开数据 baseline；
- Codex 信息摄取版本；
- 技术因子版本；
- 信息质量加权版本；
- 因子类别 ablation；
- 历史 replay matched comparison。

## 10. 当前方向的修正

项目应该从“审计很多、预测闭环弱”转向“信息摄取和预测变化可追踪”。

下一阶段优先级：

1. 建立 PredictionRunRegistry：每次预测都注册成可追踪 run；
2. 建立 InformationIntakeStore：每次信息摄取都变成结构化、本地化、可追溯的 intake；
3. 建立 MatchedPredictionDiff：每次更新都能和上一版或 baseline 做同口径对比；
4. 前端只展示最新 artifact 和关键 diff；
5. 再开始逐步强化信息源、因子 ontology、模拟精度和历史回放。

这份文档的核心提醒是：**不要让工程活动替代预测进步。**
