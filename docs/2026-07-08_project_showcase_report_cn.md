## 1. 一句话介绍

F1Predict 是一个一级方程式赛车预测研究系统。它不是简单给出“谁会赢”的黑盒答案，而是把赛道、车队、车手、练习赛、排位赛、历史成绩、天气、退赛风险、策略窗口等信息整理成一套可追溯的状态，再用模拟器生成每位车手的预计排名、冠军概率、登台概率和积分概率，并解释这些结果从哪里来。

如果用非技术语言解释，它像一个“虚拟策略分析团队”：

1. 先收集赛前能知道的信息；
2. 再判断哪些信息可靠，哪些只是弱信号；
3. 然后把信息归类到赛车、车手、赛道、天气、策略这些因素上；
4. 最后反复模拟比赛，得到概率分布；
5. 同时保留一条完整证据链，告诉我们“为什么预测会这样”。

## 3. 为什么我之前不应该执着于“稳定优势”

你最开始提到过长期愿景是希望找到稳定优势，但后续目标一直是先实现最小可用版本。当前项目不应该把“稳定优势证明”当成 MVP 终止条件。

- 当前 MVP 的验收标准是端到端可运行、可解释、可展示、可回放；
- 项目可以诚实标注“尚未证明可决策优势”，但不能因为没证明优势就说 MVP 没完成。

这点很重要，因为否则项目会被错误拖进一个非常大的金融验证问题：需要同一时间点市场价格、样本外回测、概率校准、交易成本、流动性和盘口变化。这些不是当前 MVP 的必要条件。

## 4. 项目整体设计思路

### 4.1 核心问题

普通预测模型最大的问题是：它可能直接输出一个概率，却说不清：

- 数据从哪里来；
- 哪些信息可靠；
- 哪些信息只是噪声；
- 为什么某个车手概率上升；
- 为什么某个车队被压低；
- 用户的一句话有没有被偷偷写成模型规则；
- 前端显示的到底是不是最新结果。

本项目的设计重点是解决这些问题。

### 4.2 核心原则

本项目遵循五条原则。

第一，所有预测必须有知识截止时间。  
例如预测 British Grand Prix 时，只能使用赛前能知道的信息，不能偷看赛后结果。

第二，用户反馈不能直接改变预测。  
你指出“某队明显不合理”只能触发审计和改进架构，不能被系统当成事实来源。

第三，所有信息都要归类到统一因素。  
例如“某队长距离差”不能直接写成“冠军概率下降”，而要进入赛车正赛速度、轮胎衰退、调校窗口或策略风险。

第四，模型必须保留状态更新账本。  
每条信息进入模型时，都要记录它改变了哪个状态、方向是什么、影响哪些模拟环节。

第五，预测结果必须能回放和复盘。  
模型不仅要给出当前结果，还要能在过去比赛上回放，判断它到底哪里错了。

## 5. 系统主流程

整个系统可以理解为八步：

```text
原始信息
-> 信息清洗
-> 信息质量审计
-> 统一因素分类
-> 状态更新账本
-> 比赛模拟
-> 预测结果包
-> 前端展示和赛后复盘
```

下面逐步解释。

### 5.1 原始信息

系统目前使用的信息包括：

- 官方车手积分榜；
- 官方车队积分榜；
- FastF1 正赛结果；
- FastF1 排位结果；
- FastF1 练习赛和排位赛圈速；
- 赛道官方图；
- 赛道特征向量；
- 天气和赛道温度数据；
- 市场价格快照；
- Codex/LLM 整理出的赛前证据。

其中 FastF1 是一个赛车数据工具，项目用它读取练习赛、排位赛和正赛的结构化数据。

### 5.2 信息清洗

原始数据不能直接用。系统会先做清洗，例如：

- 把不同来源里的车手名映射到统一车手编号；
- 把“被套圈但完赛”和“退赛”分开；
- 把圈速中的异常值过滤掉；
- 只使用知识截止时间之前的数据；
- 把赛道图和赛道名称对应起来。

一个很关键的修正是：系统曾经把 `Lapped` 这类“被套圈但仍完赛”的状态误当成退赛。后来已修正，避免错误污染可靠性模型。

### 5.3 信息质量审计

系统不会把所有信息一视同仁。它会判断：

- 来源是否在知识截止前；
- 来源是否有截图或归档；
- 是否来自真实结构化数据；
- 是否与其他来源冲突；
- 是否只是用户反馈；
- 是否只是弱假设。

用户反馈会被挡在默认预测之外。也就是说，你说“Leclerc 应该更强”不会直接让模型把 Leclerc 抬高。

### 5.4 统一因素分类

系统把所有信息归入几个大类。

赛道因素：

- 高速弯、低速弯、直道、超车难度；
- 赛道位置价值；
- 轮胎衰退强度；
- 安全车和红旗倾向；
- 官方赛道图和赛道向量。

赛车因素：

- 正赛速度；
- 排位速度；
- 直道速度；
- 能量回收和部署；
- 轮胎衰退；
- 可靠性；
- 升级影响。

车手因素：

- 单圈能力；
- 正赛速度；
- 保胎能力；
- 雨战能力；
- 起步能力；
- 攻防能力；
- 退赛或事故风险。

车队执行因素：

- 策略质量；
- 进站窗口；
- 调校质量；
- 比赛执行；
- 车队内部风险。

比赛环境因素：

- 降雨概率；
- 安全车概率；
- 红旗概率；
- 赛道温度；
- 轮胎压力。

### 5.5 状态更新账本

这是项目可解释性的核心。

代码里有一个“状态账本”，代码名叫 `BeliefState`。中文可以理解为：

```text
模型在某个知识截止时间下，对赛车、车手、车队、赛道和天气的当前判断。
```

例如一条信息是：

```text
某队最近三站平均完赛名次比全场更好。
```

它不会直接变成“某队冠军概率 +5%”，而是先变成：

```text
该车队赛车正赛速度状态小幅上调。
```

账本会记录：

- 来源是什么；
- 目标是谁；
- 改的是哪个因素；
- 旧状态是什么；
- 新状态是什么；
- 变化方向是什么；
- 这个变化影响模拟器哪个环节；
- 中文解释是什么。

### 5.6 比赛模拟

模拟器不是只抽一个随机冠军。它会考虑：

- 排位位置；
- 正赛速度；
- 轮胎衰退；
- 进站策略；
- 安全车窗口；
- 红旗窗口；
- 湿地比赛；
- 退赛；
- 车队同日调校窗口；
- 交通和超车难度；
- 发车和第一圈风险。

每次模拟都会得到一个完整比赛结果。重复 1200 次后，系统就能计算：

- 冠军概率；
- 登台概率；
- 进入积分区概率；
- 平均完赛名次；
- 预计积分；
- 代表性比赛回放。

### 5.7 预测结果包

每次预测会生成一个“预测结果包”，代码名叫 `PredictionPacket`。中文可以理解为：

```text
一次完整预测的证据、输入、模拟结果、解释链和审计结果的打包文件。
```

最新 British GP 预测包路径是：

```text
reports/prediction_packets_practice_conflict_gate_probe/british_gp/british_gp_20260705T000000_0000.prediction_packet.json
```

这个包里包含：

- 比赛信息；
- 数据截止时间；
- 模拟次数；
- 每位车手概率；
- 市场差异；
- 证据列表；
- 状态更新账本；
- 异常审计；
- 代表性回放；
- 前端需要展示的摘要。

### 5.8 预测运行登记

代码里还有一个“预测运行登记表”，代码名叫 `PredictionRunRegistry`。中文可以理解为：

```text
记录哪一次预测是当前最新版本，以及它对应哪个预测结果包。
```

这解决了你之前指出的一个问题：前端有时像是没更新，不知道到底显示哪一版。

现在“默认最新结果”会指向一个明确的运行记录：

```text
reports/prediction_runs/runs/british_gp/british_gp_20260705T000000_0000_20260707T122518_0000_d225707bdb.prediction_run.json
```

## 6. 可解释链条到底做到了什么

### 6.1 解释链条的目标

解释链条不是给一堆看不懂的分数，而是回答：

```text
这条原始信息为什么会影响预测？
```

完整链路应该是：

```text
原始来源
-> 提取出的事实
-> 归类后的因素
-> 状态更新
-> 模拟器受影响环节
-> 概率变化
```

### 6.2 当前实现状态

当前默认最新结果已经有完整影响追踪缓存：

```text
已覆盖状态更新数量 = 565
未覆盖状态更新数量 = 0
影响追踪记录数量 = 576
原始预测模拟次数 = 1200
追踪对照模拟次数 = 1200
追踪链条状态 = 已就绪
```

这说明 565 条状态更新都有同口径追踪，不再只是“进入了模型”，而是可以追踪它如何影响预测。

### 6.3 解释链条的中文化

过去的问题是解释里有很多裸分数，例如：

```text
race_pace +0.0352
reliability -0.0250
```

现在中文解释会尽量写成事实形式，例如：

```text
近期车队未完赛率为 0.667（6 次发车中 4 次未完赛），对比全场均值 0.273，
因此作为赛车可靠性输入，并影响退赛采样。
```

这样非技术人员也能理解：不是模型凭空给一个分数，而是从具体事实算出一个有界输入。

## 7. 针对你提出的重要问题，做了哪些改进

### 7.1 不能因为用户一句话手调数值

问题：你明确要求，用户指出“某队强/弱”只能作为问题提示，不能直接改模型。

改进：

- 用户反馈不能进入默认预测；
- 同一来源身份下的模型映射变化不能静默注册；
- 如果不是新增真实来源，而是改变模型映射，必须有模型修订证明或诊断报告；
- 未校准候选只能放在独立目录，不能覆盖前端默认展示的最新结果。

效果：

- 近期全场完赛候选没有注册；
- 近期车队可靠性候选没有注册；
- 红旗尾部候选没有注册；
- 它们都只是诊断候选，不会污染前端默认结果。

### 7.2 不再用不可解释裸分数

问题：你指出“正赛综合分”“正赛速度修正”“可靠性修正”等分数如果不解释来源，就是黑盒。代码里过去会出现一些英文内部名，比如 race score、race pace、reliability；展示时不要直接使用这些英文，而要讲清楚它们分别代表什么事实和计算过程。

改进：

- 每个结构化特征必须有中文解释；
- 状态账本必须保存来源、目标、因素、方向和影响表面；
- 前端和报告优先展示事实，而不是裸分数；
- 对可靠性、全场完赛、长距离、轮胎衰退、调校窗口等因素补了中文解释。

效果：

- 可解释模块能把“分数来自什么事实”说出来；
- 解释链从“模型认为”变成“某来源事实导致某状态变化”。

### 7.3 赛道图不正确

问题：早期前端有不真实或不完整的赛道图，British GP 的 Silverstone 也被指出有问题。

改进：

- 改用 F1 官方赛道图资产；
- 加入赛道图审计；
- 前端标注官方底图来源；
- 对旧图和模拟叠加做了清理。

当前前端显示：

```text
官方赛道图 + 模拟回放叠加层
来源类型：F1 官方赛道图
视觉审计状态：已验证
赛道：银石
采集时间：2026-07-01 10:45:03
```

### 7.4 模拟回放不再不可用

问题：你指出前端曾经显示“模拟回放不可用”。

改进：

- 预测包保存代表性模拟回放；
- 前端读取回放行数据；
- British GP 当前有 312 行 replay 数据；
- 页面显示赛道图和回放叠加层。

### 7.5 前端显示太多、重点不清

改进：

### 7.6 “车”的重要性没有体现

问题：你认为模型低估了赛车/车队实力，过度依赖车手层信号。

改进：

- 增加车队强度重估；
- 增加全场完赛顺位候选；
- 增加近期车队可靠性候选；
- 增加练习赛车队长距离信号；
- 增加车队轮胎衰退状态；
- 增加车队调校窗口状态；
- 增加同队共享比赛日窗口波动。

结果：

- 这些通路都能追溯到来源；
- 它们能改变预测概率；
- 但当前默认最新结果仍没有完全修好“梅赛德斯偏高、法拉利和勒克莱尔偏低”的排序问题。

### 7.7 最近几站状态没有充分影响

问题：你指出模型应该看预测比赛前几站的情况，而不是只看长期静态先验。

改进：

- 使用最近窗口成绩；
- 使用赛季累计成绩；
- 使用近期相对更早窗口的趋势；
- 使用全场完赛顺位，不只看积分区；
- 使用近期未完赛率进入可靠性。

诊断候选效果：

```text
当前默认最新结果:
Russell 47.75%，Antonelli 43.75%，Hamilton 5.33%，Leclerc 1.58%

近期车队可靠性候选:
Antonelli 39.08%，Russell 38.75%，Hamilton 9.17%，Leclerc 4.25%
```

这说明近期状态通路确实能改变结果，但还没有通过校准门槛。

### 7.8 前端加载慢和缓存问题

问题：你指出前端加载慢，且显示状态像是没有更新。

改进：

- 引入预测运行登记；
- 前端默认读取最新注册预测包；
- 完整影响追踪拆成独立缓存；
- 前端不再必须把所有影响追踪都塞进主结果包；
- 最新影响追踪缓存覆盖 565/565 条状态更新。

仍需改进：

- 前端还不是独立前后端工程；
- 仍建议后续生成专门的前端缓存文件，让页面只读缓存，不实时计算。

## 9. 技术架构说明

### 9.1 数据层

数据层负责读取和整理原始信息。

主要文件：

```text
src/f1predict/results.py
src/f1predict/session_laps.py
src/f1predict/official_standings.py
src/f1predict/market.py
src/f1predict/track_features.py
src/f1predict/weather_profiles.py
```

中文解释：

- `results.py`：读取正赛和排位结果；
- `session_laps.py`：读取练习赛、排位赛圈速并做摘要；
- `official_standings.py`：读取官方积分榜；
- `market.py`：读取市场价格；
- `track_features.py`：生成赛道特征；
- `weather_profiles.py`：整理天气资料。

### 9.2 信息摄取层

信息摄取层负责把网页、新闻、报告、采访等非结构化信息变成结构化证据。

主要概念：

- 来源候选：先找到可能有用的网页或资料；
- 研究包：把来源、摘要、证据和质量记录放在一起；
- 证据声明：把一句信息转成“谁、什么因素、正负方向、可信度”；
- 质量审计：判断来源是否可靠、是否在知识截止前、是否和其他信息冲突。

代码中有一些英文名，展示时可以这样解释：

- `EvidenceClaim`：证据声明；
- `EvidenceQuality`：证据质量评分；
- `factor contract`：因素合同，规定哪些信息能进入哪些因素；
- `factor trace`：因素路由记录，说明信息会影响哪个模拟环节。

### 9.3 状态层

状态层是项目核心。

代码名：

```text
BeliefState
```

中文解释：

```text
状态账本，记录模型在赛前某一时刻对赛车、车手、车队、赛道、天气的判断。
```

它包含：

- 赛车状态；
- 车手状态；
- 车队执行状态；
- 比赛环境状态；
- 来源列表；
- 提取信息；
- 归一化证据；
- 质量评分；
- 状态更新账本。

### 9.4 模拟层

模拟层负责把状态变成比赛概率。

主要文件：

```text
src/f1predict/models/pace.py
src/f1predict/models/simulator.py
```

中文解释：

- `pace.py`：把赛车、车手、赛道和状态信息转成速度评分；
- `simulator.py`：模拟比赛，包括发车、进站、轮胎衰退、安全车、红旗、湿地、退赛和最终排名。

注意：报告中不要直接说“race score”，而应解释为：

```text
模型内部的正赛速度估计值，用来把状态因素转成每圈时间差。它不是一个独立事实，而是多个来源和状态共同计算出的内部量。
```

### 9.5 预测包和登记层

主要概念：

- 预测结果包：一次预测的完整结果；
- 预测运行登记表：记录哪个预测包是最新版本；
- 预测差异：比较两个预测版本的输入和输出差异。

对应代码名：

- `PredictionPacket`：预测结果包；
- `PredictionRunRegistry`：预测运行登记表；
- `MatchedPredictionDiff`：同口径预测差异比较。

中文解释：

`PredictionRunRegistry` 的作用是解决“前端到底显示哪一版”的问题。  
`MatchedPredictionDiff` 的作用是比较两个预测包，判断变化来自新来源、模型修订还是缓存不同步。  
这些概念不是为了增加复杂度，而是为了防止模型结果偷偷变化却说不清原因。

### 9.6 前端层

主要文件：

```text
web/app.js
web/styles.css
src/f1predict/api_v2.py
src/f1predict/server.py
```

中文解释：

- `server.py`：本地服务器；
- `api_v2.py`：给前端提供预测包、最新运行、影响追踪等接口；
- `app.js`：前端页面逻辑；
- `styles.css`：页面样式。

当前前端不是独立大型前端工程，而是本地服务同时提供页面和接口。对 MVP 展示足够，但后续正式产品应拆成更清晰的前后端缓存架构。

## 12. 明天展示建议

### 12.1 给非技术人员的 3 分钟讲法

可以这样讲：

```text
我们做了一个 F1 预测研究系统。它不只是给一个冠军概率，而是把赛前能知道的各种信息整理成赛车、车手、赛道、天气、策略等因素，再模拟很多次比赛。

最重要的是，它能解释为什么这么预测。每条信息从哪里来、如何影响模型、最后让哪个车手概率变了，都可以追踪。

当前项目已经完成最小可用版本：能跑预测、能看赛道图、能看模拟回放、能看解释链、能做赛后复盘。

但它还不是成熟预测模型。比如 British GP 预测仍偏向 Mercedes，Leclerc 概率偏低。我们现在能诚实看到这个问题，并且能定位问题来自哪些模型环节，这就是这个 MVP 的价值。
```

### 12.2 给技术人员的 5 分钟讲法

可以这样讲：

```text
系统以知识截止时间为边界，读取结构化数据和 Codex 整理的证据。这里的 Codex 可以理解为“把非结构化网页、新闻、采访整理成本地证据的助手”。所有证据先进入统一因素体系，再更新状态账本。状态账本记录每个因子的旧值、新值、来源、置信度和影响位置。

模拟器读取状态账本，生成排位、正赛速度、轮胎衰退、策略、安全车、红旗、湿地和退赛等分支。预测结果写成“预测结果包”，再登记到“预测运行登记表”。前端只读取登记表里标注为“最新”的那一次运行。

可解释性通过“同种子影响追踪”实现：对每条状态更新做隔离重跑，比较它对预测概率的影响。当前默认最新结果已有 565/565 条状态更新覆盖，来源模拟次数和追踪模拟次数都是 1200。

系统还内置门禁：用户反馈不能直接入模；模型映射变化不能静默注册；未校准候选只能作为“诊断候选”，不能覆盖前端默认结果。
```
